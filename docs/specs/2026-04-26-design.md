## 1. Contexto e motivação

A SEA Tecnologia tem trilha formal de onboarding para devs backend Liferay com 3 desafios obrigatórios (30/60/90 dias) sobre o **Hotel Livingstone** (case fictício). Em paralelo, há a trilha de carreira (Junior I/II/III, Pleno I/II/III, Sênior I/II/III) com desafios próprios.

Felipe está **classificado como Junior II** na SEA desde abril/2026, vindo de background fullstack Django/Python (~2-3 anos) com Java pontual (Spring Boot, Java 8). Liferay é stack nova — primeira task entregue (#45829, GDF Sprint I) em 2026-04-23.

**Objetivo do projeto:** entregar os 3 desafios obrigatórios (30/60/90) num único produto evolutivo (Hotel Livingstone), com qualidade técnica de **Pleno** (não Sênior cheio — vide análise honesta na seção 3), enquanto reforça fundamentos via aprendizado guiado.

**Por que num projeto único:**
- Narrativa coerente do M1 ao M3 (mesmo domínio cresce em camadas técnicas)
- Cobre Junior I/II/III por tabela (sem precisar de projeto separado)
- Material acumulado serve futuras entrevistas e workshops Pleno I/II

**Por que não "Sênior cheio":**
- Sênior real exige liderança técnica + workshops + mentoria + experiência em produção
- Trilha SEA Pleno I/II/III tem requisito formal de **workshop gravado** (incluído via Upgrade C)
- Sênior I/II/III SEA exige ministrar workshops + auxiliar mentorandos — não simulável em projeto solo
- **Análise honesta:** este projeto entrega "Junior III muito difícil de ignorar" + abre caminho pra Pleno via workshops

## 3. Escopo

### Dentro do escopo
- Site Hotel Livingstone (Pages, Fragments, Web Content, Theme, Navigation, Multi-idioma PT/EN)
- Dynamic content (Web Content Structures+Templates, DDL)
- Workflow Kaleo (Solicitação Reserva → Aprovação → Notificação)
- Migração Liferay nativo → Docker Compose (Liferay + Postgres + Elasticsearch)
- Service Builder: domínio do hotel (Room, Booking, Guest, Package) com finders e validações
- REST Builder: API headless do hotel
- TDD em camada de domínio (services + ResourceImpl)
- Tests unit + integration
- CI/CD GitLab (lint, build, tests, format-check)
- ADRs documentando decisões técnicas
- Retrospectivas por milestone
- 3 apresentações com slides (markdown export)
- **Upgrade C:** Workshop testes gravado + Liferay Objects + diagnóstico real

### Fora do escopo (backlog "se sobrar tempo")
- Cobertura de testes ≥80% (alvo realista: 50-60%)
- Observabilidade profunda (Prometheus/Grafana, distributed tracing, correlation IDs)
- Performance benchmark formal (load testing com k6/JMeter)
- Security depth (pen testing OWASP completo, secrets rotation)
- Refactor cycles (refazer features depois de aprender padrões melhores)
- Frontend custom além do que o tema/fragments cobrem (sem React/Vue separados)
- Integração com sistemas externos reais (CRM, payment gateway, etc.)

### Explicitamente NÃO faz parte
- Cursos formais Liferay University (pode ser feito em separado se RH exigir certificado)
- Workshops Pleno III, Sênior I/II/III (workshops formais separados, fora do projeto)
- Mentoria de outros devs (requisito de trilha Sênior, não simulável solo)

## 7. Loop de aprendizado (o "curso guiado")

5 mecânicas que rodam em paralelo durante todo o projeto:

### 7.1. Sessão Q&A com Claude
Formato que firmou na consolidação pós-Task 1 do GDF:
- **Pergunta** curta de mim (Claude)
- **Tentativa** do Felipe
- **Correção cirúrgica** ≤5 linhas
- **Mini-resumo** 2-3 linhas
- **Próximo conceito** (1 por vez, sem empilhar)

Princípios já internalizados nas memórias do projeto:
- 1 conceito por vez
- Ação concreta > teoria abstrata
- Analogias Python/Django só quando solicitadas
- Pausa explícita antes de avançar
- Preferir codar a teorizar

### 7.2. Learning log por sessão
**Onde:** `docs/learning-log/YYYY-MM-DD-titulo.md`
**Formato (~20 linhas):**
```markdown
# YYYY-MM-DD — <título da sessão>

## O que tentei
...

## O que errei
...

## O que firmou
...

## Dúvida aberta
...
```
**Custo:** ~10min ao final de cada sessão. Vira matéria-prima das retrospectivas.

### 7.3. ADR como instrumento pedagógico
**Estrutura obrigatória** (não só "registrar a decisão"):
1. **Contexto** — o que motivou a decisão
2. **Opções consideradas** — pelo menos 2-3 alternativas com prós/cons
3. **Trade-offs** — o que ganhei e o que perdi
4. **Decisão** — qual foi a escolha
5. **Consequências** — o que isso implica daqui pra frente

A seção "Opções consideradas" é o gancho pedagógico — força defender por que NÃO escolheu as outras. É aí que o aprendizado real acontece.

### 7.4. Retrospectiva por milestone
**Formato:** igual `~/Documents/SEA/roadmap-estudos-liferay/task-1-rest-builder-retrospectiva.md`
**Tamanho:** ~500-600 linhas
**Quando:** final de M1, M2, M3 (antes do commit final do milestone)
**Estrutura:**
1. Contexto
2. Preparação técnica
3. Decisões (aplicadas / descartadas / desvios)
4. Timeline de execução em subseções numeradas
5. Auto-avaliação de boas práticas
6. Artefatos produzidos
7. Status e pendências
8. Próximas tasks
9. Dúvidas pro time
10. (se houver) Code review aplicado

**Por que esse tamanho:** vira (a) base das slides de apresentação, (b) material consultável em entrevista futura, (c) auto-avaliação honesta.

### 7.5. "Refaz até ficar bom"
**Princípio:** primeira tentativa feia = refaz. Não passa pra próxima feature só pra cumprir cronograma.
**Critério objetivo de "ficou bom":** Felipe consegue explicar a decisão sem me consultar.
**Margem:** os 20% de folga calculados em cada milestone existem exatamente pra absorver re-trabalho.

**Risco do princípio (perfeccionismo):** mitigado pelo critério objetivo acima. Se já consegue explicar, segue. Não precisa estar "perfeito".

## Apêndice B — Glossário rápido (termos Liferay que aparecem no spec)

- **ADR (Architecture Decision Record):** documento curto registrando decisão técnica + alternativas
- **Asset Publisher:** widget Liferay que lista conteúdo dinamicamente baseado em filtros
- **Blade CLI:** ferramenta Liferay pra scaffolding e build
- **DDL (Dynamic Data Lists):** listas de dados estruturados criadas via UI Liferay
- **Fragments:** componentes UI reutilizáveis criados via Fragment Editor
- **Kaleo:** engine de workflow nativo do Liferay
- **Liferay Objects:** feature low-code do 7.4 que substitui Service Builder em casos modernos
- **OSGi:** runtime modular usado pelo Liferay (cada bundle é um módulo independente)
- **REST Builder:** ferramenta Liferay que gera API headless a partir de OpenAPI YAML
- **Service Builder:** ferramenta Liferay que gera entidades JPA + services + finders a partir de XML
- **Vocabulary/Categories:** taxonomia Liferay (similar a tags hierárquicas)
- **Web Content Structure+Template:** schema + apresentação de conteúdo dinâmico no Liferay
