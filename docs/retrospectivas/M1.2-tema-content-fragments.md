## Decisões aplicadas vs. desvios do plan

**Aplicadas tal como planejado:**

- Brainstorm 2026-04-29 cravou 4 decisões fundamentais (tema + fragment com escopos, parent Styled, template js-theme, ordem top-down) e todas as 4 sobreviveram intactas até o fim do M1.2
- Princípio "quem edita = onde mora; estrutura ≠ conteúdo" aplicado consistentemente — cada artefato tem dono claro (dev edita tema, content creator de layout edita fragment, content creator de dados edita Web Content)

**Desvios conscientes:**

- **Selection múltipla > Text Repeatable** pra campo amenities da Structure Quarto. Plan original sugeria Text Repeatable; durante a Fase 2 escolhi Selection múltipla (vocabulário fechado, consistência > flexibilidade). Side-effect: `${field.getData()}` retorna labels no idioma corrente (não keys), e POST `/structured-contents` via Headless Delivery falha com `RequiredValue` em Selection múltipla — bloqueador potencial pro Site Initializer M2 (mitigação possível: Service Layer Java, LAR import, ou PUT replace)
- **Tema custom como Site default → trocado pra override per-page (Fase 4).** Plan assumia tema como default global; descobrimos que isso quebra Page Editor + Site Admin (ADR-006). Decisão revisada com ganho líquido: Site Admin permanece funcional, identidade aplicada onde importa
- **Pages criadas no M1.1 deletadas e recriadas (Fase 4).** Layout default do wizard tinha item órfão (Fragment Entry Link apontando pra fragment removido) que crashava Page Editor com "Erro inesperado ao renderizar este item". Workaround único viável: deletar + recriar com template "Vazio"
- **Articles "Quarto" — nomes Drácula em vez de Safari.** Plan original sugeria Suíte Kilimanjaro/Serengeti/Okavango (tema safari). Durante Fase 1 a paleta Dracula+Apple emergiu como identidade — coerência forçou trocar pra Carmilla/Lestat/Drácula. Decisão fechada 2026-05-01

## O que aprendi (resumo dos learning-logs)

**Tema custom + parent Styled (Fase 1)**

- Generator-liferay-theme 10.2.0 não pede prompts interativos — gera com defaults (parent Styled coincidiu com Decisão #2 sem precisar escolher). Validação pendente: existe flag pra forçar prompts?
- Em workspace, **fluxo Gulp é desnecessário** pro build/deploy do dia-a-dia — Gradle plugin do workspace constrói/deploya tudo. `gulp init` skip; `liferay-theme.json` no `.gitignore`
- `build/templates/portal_normal.ftl` (gerado pelo build) tem a versão herdada do parent Styled — fonte mais confiável pra customizar do que escrever do zero
- `<#include init />` no topo do FTL é essencial — popula `nav_items`, `themeDisplay`, locale, etc. Não pode tirar
- `nav_items` aponta automaticamente pro Primary Navigation — não precisa buscar por nome. Configuração da Task 8 do M1.1 entrou em prod sem código adicional
- `aria-current="page"` precisa ser setado manualmente via `nav_item.isSelected()` — não vem auto
- **Bug `themeDisplay.getURLHome()` retorna `/web/guest/` (HTTP 404)** — `groupName` interno persiste como "guest" mesmo após renomear UI pra "Hotel Livingstone". Fix: URLs relativas (`/sobre`, `/contato`). Pendência M2 multi-site
- SDKMAN auto_env só dispara na pasta exata onde mora `.sdkmanrc` — `cd workspace` direto pega Java global (11.0.25-tem do GDF). Sempre passar pela raiz primeiro
- Conflito ES sidecar GDF×Hotel na 9200/9201 — parar GDF antes de subir Hotel (Liferay carrega Elasticsearch interno)

**Web Content (Fase 2)**

- **3 "templates" do Liferay são coisas diferentes:** Web Content Template ≠ Display Page Template ≠ Application Display Template. Os 3 são DDMTemplate por baixo, mas com `classNameId` diferente
- `${field.getData()}` em Selection múltipla retorna **labels** no idioma corrente (não keys). JSON do Article tem campo dual: `data` (labels) + `value` (keys). Comportamento NÃO documentado claramente — descoberto via Headless API empiricamente
- Dictionary `amenityLabels` no Template "Quarto Default" é **redundante** — refactor pendente: trocar pra `<#list amenityKeys as label><li>${label}</li></#list>` (M1.3 ou polimento futuro)
- **Headless Delivery POST `/structured-contents` falha em Selection múltipla** — erro genérico `DDMFormValuesValidationException.RequiredValue` "No value defined for field name" mesmo com payload idêntico ao GET. Bloqueador potencial pro M2 Site Initializer
- **Web Content Display widget em Widget Page** = caminho canônico pra testar Template em pages do hotel (Content Pages não suportam widgets diretos). Page `/teste` reusada nas Fases 3/4
- Rich Text quer HTML/toolbar, não markdown — `**bold**` colado fica literal. Pra colar HTML preparado: usar Source view do CKEditor (`<>` button)

**Fragments (Fase 3)**

- **Click "Adicionar X" + Enter** funciona em page vazia (Liferay 7.4 a11y mode pra dnd inacessível). Falha em page com layout pré-existente porque "primeiro target válido" fica ambíguo
- **Drag-and-drop sintético via Playwright/CDP NÃO funciona** com react-dnd HTML5 backend do Liferay 7.4 GA132 — só drag REAL do mouse OU click+Enter via teclado
- Page criada via UI tem layout default Contêiner > Grade 2-col com "Bem vindo ao Liferay" se for Content Page. Pra page completamente vazia, criar nova Content Page do scratch (não a Home)
- Mapping `card-photo` requer aba **Navegador** (não click direto na imagem que abre Document Picker). Painel direito muda pra abas "Mapear / Link" (Mapear = pra Article, Link = pra URL custom)
- **Critério canônico do M1.2 passou:** alterar título Article "Suíte Carmilla" → "REVISADA" → page refletiu sem republicar fragment. **Mapping = runtime, Article = fonte de verdade, fragment = apresentação**
- `data-lfr-editable-type="rich-text"` dá toolbar inline (B/I/link) no Page Editor; `text` é plain text simples; `image` abre Document Picker; `link` abre URL chooser

**Composição + override de tema (Fase 4)**

- **Page Editor pode crashar** se layout default tem item órfão (Fragment Entry Link apontando pra fragment removido). Workaround: deletar+recriar a page com template "Vazio"
- **Drag-and-drop fragments via Chrome DevTools MCP é não-confiável.** Hero+Galeria inseriram via click programático + Enter; Card de Quarto não inseriu por nenhum método. Drag REAL do mouse continua sendo o caminho oficial pra automação
- **Tema custom como Site default = QUEBRA Page Editor + Site Admin.** Faltam templates pra contextos internos (control_panel, configuration views, iframe Page Editor). Solução canônica: Site default = Classic + override `hotellivingstonetheme` per-page via UI Configurar Página → Look and Feel
- **API JSONWS pública NÃO ativa override per-page.** `/api/jsonws/layout/update-look-and-feel` só seta `themeId` — não ativa o flag interno `useThemeSettings=true` (coluna SEPARADA na tabela Layout, não exposta via JSONWS). Tentar via `update-type-settings` adicionando `useThemeSettings=true` em typeSettings também não funciona. **Override per-page = via UI obrigatório**
- **JSONWS Layoutset funciona pra config site-wide reversíveis** — POST `/api/jsonws/layoutset/update-look-and-feel` com `themeId=hotellivingstonetheme_WAR_hotellivingstonetheme` cobre TODAS as pages com herança em 1 chamada. Bom pra rollback rápido
- "Web Content Display" virou **"Publicador de Conteúdo Web"** em Liferay 7.4 GA132 PT-BR. Está em aba Widgets (não Fragmentos). Versão simplificada em DESTAQUES, completa em GERENCIAMENTO DE CONTEÚDO
- **Mapping de fragments aceita Article do mesmo site** sem precisar de Display Page Template — visual no editor mostra "Item: Suíte Lestat / Tipo: Artigo de Conteúdo Web / Subtipo: Quarto" e dropdown popula com Title/Subtitle/Description/Price/Comodidades da Structure custom
- Dropdown de Campo do mapping tem 2 seções: **"Informação Básica"** (metadata Liferay: Title/Description do Article) vs **"Conteúdo (Quarto)"** (campos da Structure). **Sempre preferir Conteúdo (Quarto)** — semântica explícita

**i18n (Fase 5)**

- Display name de Page é i18n-aware por **atributo** (1 valor por locale), não por registro inteiro — não duplica a page, só adiciona traduções no form Configure Page → Translations
- **Friendly URL no Liferay 7.4 GA132 é Localized Friendly URL** — i18n por locale igual ao display name. Cada page tem 1 friendly URL por locale ativo no site (ex: `/quartos` em PT-BR + `/rooms` em EN-US, ambas canônicas, ambas HTTP 200 sem redirect). Configurado via Configure Page → bandeira de locale no field de Friendly URL, mesmo padrão de UI da tradução de display name
- **Decisão tomada na Fase 5:** traduzir display name **e** friendly URL pras 5 pages do menu. Resultado: cada locale tem URL canônica nativa (`/pt-BR/quartos` + `/en-US/rooms`) + display name no idioma correto + zero redirects. Padrão i18n completo
- **Achado intermediário interessante** (durante a investigação antes do fix): se você renomear friendly URL **sem ativar i18n por locale** (ex: trocou só o valor canônico pra `rooms`), o Liferay mantém o nome antigo (`quartos`) como **alias 301** automático apontando pro novo. Útil pra preservar link equity quando friendly URL muda em produção. Comportamento confirmado empiricamente nessa retrospectiva — não bloqueante pra Hotel Livingstone porque optamos pela solução completa (Localized Friendly URL), mas vale doc pra futuras renomeações em prod

## Dificuldades encontradas

- **Page Editor crashando em layout default do wizard** (Fase 4) — sintoma confuso ("Erro inesperado ao renderizar este item") sem stack trace útil. Solução veio por tentativa empírica de deletar + recriar. Lição: Liferay 7.4 a11y mode + react-dnd HTML5 backend = qualquer suposição precisa validação empírica
- **Tema custom como Site default quebrando Site Admin** (Fase 4) — sintoma assustador (sidebar de configuração render quebrado), causa não óbvia (templates internos de control_panel ausentes no parent Styled). Investigar e formalizar virou ADR-006
- **Mapping de imagem do Card de Quarto** (Fase 3) — click direto na imagem do fragment abre Document Picker (escolher arquivo), não abre painel de mapping. Solução: aba Navegador da sidebar esquerda → click no nó `card-photo`. Não óbvio até descobrir
- **Selection múltipla bloqueando Headless POST** (Fase 2) — POST `/structured-contents` recusa payload idêntico ao GET com erro genérico. 1h investigando antes de aceitar como bloqueador conhecido
- **Drag-and-drop via MCP não-confiável** (Fase 3 e 4) — Chrome DevTools MCP consegue Hero + Galeria via click programático + Enter, mas Card de Quarto não inseriu por nenhum método sintético. Felipe finalizou via drag REAL no browser dele. Lição: react-dnd HTML5 backend exige KeyboardEvent genuíno (humano), não sintético
