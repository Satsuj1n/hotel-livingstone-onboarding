## O que aprendi (resumo dos learning-logs)

**Doc-first + ADR**

- Doc-first não é burocracia: serve como spec executável que o próprio dev consulta enquanto trabalha.
- ADR no formato "Contexto → Opções → Trade-offs → Decisão → Consequências" força raciocinar contra a decisão antes de tomá-la — defender por que NÃO escolheu B é mais útil que justificar C.

**Workspace Liferay**

- `blade init -v 7.4 workspace` é o equivalente Liferay do `django-admin startproject` — gera Gradle multi-projeto com `modules/`, `themes/`, `configs/`, `gradlew`.
- `liferay.workspace.product` em `gradle.properties` define a versão alvo. Default do Blade em 2026-04 = `portal-7.4-ga132` (3 patches à frente do GDF).
- Template moderno (2026) injeta config pra 6 IAs (`.cursor/`, `.gemini/`, `.windsurf/`, `.claude/`, `.github/copilot-*`, `.workspace-rules/`). Apaguei todas — regras steering pra Client Extensions conflitavam com objetivos do M3 (Service Builder/REST Builder).
- `.gitignore` com separation por escopo: `workspace/.gitignore` cobre Gradle/IDE/build internos; raiz cobre OS/credenciais/AI tooling residual.

**Bundle + servidor local**

- GA132 extrai Tomcat em `bundles/tomcat/` (sem versão no nome). Diferente de versões antigas e do que o plan original assumia (`tomcat-*/`).
- `blade server init` reaproveita ZIP de cache em `~/.liferay/bundles/` quando já existe — init em ~37s em vez de ~10min de download.
- `blade server run` (foreground) preferível a `start` (background) em dev local — vê os logs subindo em tempo real.

**Setup Wizard (admin)**

- Setup Wizard vem **desabilitado por padrão no bundle GA132** (`setup.wizard.enabled=false` em `portal-setup-wizard.properties`). Pra forçar: setar pra `true` + apagar `bundles/data` + `bundles/osgi/state` + `bundles/work` + restartar.
- `portal-setup-wizard.properties` é **write-back** — Liferay reescreve o arquivo com os valores preenchidos após Finish Configuration + desliga a flag de novo. Útil pra reproduzir o setup em outra máquina copiando esse arquivo.
- Fluxo real do wizard no GA132: **Basic Configuration → Termos de Uso → Criação da senha do admin → Home logado**. A senha NÃO é `test` se passar pelo wizard completo.
- Esquecer senha do admin = resetar instância inteira (HSQL não tem hash recover trivial).

**Site + Pages**

- Setup Wizard **cria automaticamente um Default Site** usando o "Nome da empresa" do setup. Não precisei criar — só configurar.
- Default Site nasce com friendly URL `/guest` independente do display name — mudei manualmente pra `/hotel-livingstone`.
- **Content Page tem workflow draft → published** no Liferay 7.4+. Criar a page não publica — fica como `RASCUNHOS` até clicar em Publicar. Sem publicar, `curl` retorna 404.
- "Vazio" em "MODELOS BÁSICO" = Content Page (modelo moderno do 7.4+); "Página de Widget" em "OUTRO" = tipo clássico (drag-and-drop de portlets).
- Default Landing Page é implícito no GA132 — acessar `/web/hotel-livingstone` direto renderiza a primeira page da lista. Plan original pedia "Set as Default Landing Page" explicitamente, não foi necessário.
- Search Page (`/search`) é criada automaticamente pelo core — built-in, com tradução nativa (PT: "Buscar", EN: "Search").

**i18n no Liferay**

- i18n tem **3 níveis**: Instance / Site / Web Content. Ativei no escopo do Site porque Hotel Livingstone tem público próprio — Instance afetaria sites futuros que talvez só sirvam PT.
- A UI "Atual / Disponível" inverte a intuição: "Atual" = idiomas ATIVOS no site, "Disponível" = idiomas instalados na Instance mas inativos neste site.
- `I18nFilter` parseia o primeiro segmento da URL como locale: sem prefixo = fallback no Default Language (HTTP 200); locale ativo = serve nesse idioma (HTTP 200); locale inativo/inválido = **HTTP 404 explícito**.
- Análogo Django: `LocaleMiddleware`. Liferay diferencia "ausência de prefixo" (válido, fallback) de "prefixo de locale não suportado" (erro 404) — por design SEO/deep-link.
- **Friendly URL é única e estável por page** — independe de locale. O que muda é o **display name** (i18n-aware).
- **Ativar idioma ≠ traduzir conteúdo.** As 6 pages têm display name só em PT-BR — pendência pra M1.2.

**Navigation Menu**

- Navigation Menu é **entidade independente das Pages**, com relação N:M. Padrão Composition over Coupling — site pode ter N menus (Header, Footer, Sidebar), 1 page em vários menus, items que não são pages (URL externa, custom).
- **Primary Navigation conecta o menu ao slot do tema.** Sem marcar como Primary, o menu existe no banco mas o tema não o renderiza. O slot expõe ao tema via FreeMarker (`nav-items` no Classic).
- Permissioning é resolvido em runtime separadamente — items apontando pra page sem permissão View pro user atual são filtrados automaticamente. Separação Page/Menu existe pra **expressar intenção de UX**, não pra fazer access control.
- Liferay 7.4 GA132 gera URLs curtas pro Default Site (`/quartos`) em vez de `/web/hotel-livingstone/quartos`. Ambas roteiam corretamente, mas URL curta **só** funciona pro Default Site da Instance — sites adicionais (M2 multi-site) sempre exigem `/web/<friendly-url-site>/<page>`.

## Dificuldades encontradas

Tudo bem tranquilo no geral — os atritos foram mais comandos incorretos / suposições de plan que o GA132 desmentiu do que dúvidas técnicas reais.

- **`sed` com glob `bundles/tomcat-*/conf/server.xml` falhou silenciosamente** porque GA132 extrai o Tomcat em `bundles/tomcat/` (sem versão no nome). Resolvi manualmente abrindo o `server.xml` no editor e fazendo Replace All 8080 → 8081. Lição meta: comando com glob (`*`) precisa ser validado contra a estrutura real antes de mandar pro terminal.
- **Plan supôs que o Site "Hotel Livingstone" e a Page "Home" não existiam** — mas o Setup Wizard cria os dois automaticamente usando o "Nome da empresa". Aproveitei o que estava lá em vez de duplicar.
- **Setup Wizard veio desabilitado por padrão no bundle GA132** — atrito de descoberta, não bloqueio. Reativei via property + limpeza das 3 pastas.
- **Validação curl do menu retornou 0 ocorrências dos labels na primeira tentativa** porque busquei pelo padrão `>Label<` (texto colado entre tags). Liferay envolve labels do menu em `<a class="nav-link..." href='...' role="menuitem">\n\t\t\tLabel\n</a>` — texto vem após whitespace e quebras. Lição: ao validar HTML renderizado, usar regex permissiva (`grep -oE 'Label'`) primeiro, antes de assumir que falta algo.

## Pendências carregadas pra M1.2

- **Site configurado via UI não está versionado** (decisão pra M2: Site Initializer).
- **Edição manual de `server.xml`** após `blade server init` é manual a cada init de bundle novo — documentado no README.
- **6 pages têm display name só em PT-BR.** Em EN-US renderizam o nome PT mesmo (`<title>Quartos - Hotel Livingstone</title>` ao acessar `/en-US/web/hotel-livingstone/quartos`). Traduzir display names individualmente é trabalho de Content Creator que entra em M1.2.
- **URLs curtas (`/quartos`) hardcoded em conteúdo** vão quebrar quando a instância tiver múltiplos sites e Hotel Livingstone deixar de ser Default. Boa prática: sempre usar friendly URL longa ou macro do tema.
